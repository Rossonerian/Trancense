# Trancense

Trancense is a pitch-ready, evidence-led AI energy audit platform MVP for Indian industrial and commercial facilities. It provides decision support for auditors, facility managers, reviewers, and executives without pretending to be a statutory certification, compliance, or control system.

The seeded demo is **Shakti Precision Components Pvt. Ltd. — Pune Plant**, an illustrative Detailed audit for `01 Apr 2025 — 31 Mar 2026`.

## Product boundaries

The application is a deterministic engineering workflow and reporting prototype. It does not provide BEE certification, PAT filing, legal compliance approval, automatic equipment control, predictive-maintenance failure prediction, a digital twin, subsidy/net-metering eligibility, vendor endorsement, live Modbus/BACnet/MQTT/SCADA/PLC/BMS connections, native mobile apps, or DOCX/XLSX export.

Numerical outputs are generated by typed pure functions in [`lib/calculations.ts`](./lib/calculations.ts). Every material demo result carries a calculation/source ID, unit, status, confidence grade, assumptions, reporting period, and formula version. Demo tariffs, benchmarks, emission factors, production data, and solar assumptions are illustrative and versioned metadata, not commercial or regulatory advice.

## Stack and architecture

- Next.js App Router 16, React 19, strict TypeScript, and CSS design tokens.
- Accessible Lucide icons and responsive dark editorial SaaS UI; no external map or telemetry integration.
- Shared demo adapter in [`lib/demo-data.ts`](./lib/demo-data.ts), with tenant-scoped domain types in [`lib/types.ts`](./lib/types.ts).
- Pure, independently testable calculation engine for energy, reconciliation, demand, equipment savings, finance, emissions, and solar.
- Client-only interaction components for CSV validation demo, analytics filters, ECM ranking/status editing, solar scenarios, grounded assistant prompts, and print action.
- The current persistence layer is deterministic in-memory demo data. Repository/database interfaces can be introduced behind the same domain types when production persistence is needed.

Routes:

`/overview` · `/audits/[id]` · `/data` · `/assets` · `/energy-balance` · `/analytics` · `/ecms` · `/solar` · `/reports` · `/assistant` · `/settings`

## Local setup

Requirements: Node.js 20+ (Node 24 is also supported) and npm.

```bash
git clone https://github.com/RossoKashy/Trancense.git
cd Trancense
npm install
cp .env.example .env.local
npm run dev
```

Open [http://localhost:3000](http://localhost:3000). No database, account, API key, or external service is required for demo mode.

Demo entry: open `/overview`. The shell is intentionally frictionless and shows the active **Demo Data** badge. The demo role is presented as `Energy Auditor / Ananya Rao` for the seeded Pune Plant audit.

## Environment variables

All variables are optional for the MVP:

- `AI_PROVIDER=demo` keeps the deterministic grounded fallback active.
- `AI_API_KEY` is reserved for a future server-side provider adapter and is never needed by the demo.
- `DATABASE_URL` is reserved for future PostgreSQL persistence; it is not read by the current demo adapter.

Never commit `.env`, `.env.local`, credentials, private keys, or uploaded data. Use `.env.example` for names and safe comments only.

## Verification

```bash
npm run typecheck
npm run lint
npm test
npm run build
```

The focused calculation tests cover zero/negative inputs, five-percent reconciliation tolerance, pump/fan and lighting savings, payback/NPV/IRR failure behavior, and solar capacity/loss/degradation behavior. The production build statically generates all fixed routes and server-renders the dynamic audit route.

The repository does not currently include Playwright. The smoke path is implemented through the linked UI flow: **Open demo → view audit → inspect energy balance → change ECM ranking weights → open report**. Add Playwright when browser automation is part of the deployment environment.

## Calculation and AI principles

Calculations use `calc-2026.01`, with explicit `MSEDCL-demo-v2` tariff metadata and `India-grid-CEA-demo-2025` illustrative emission-factor metadata. Invalid inputs are clamped or return a safe zero/null result; display rounding occurs only at the UI boundary. ECMs with interaction groups are shown together for screening but flagged for combined modeling before approval.

Without a provider key, `/assistant` uses a deterministic **Grounded Demo Response** assembled only from seeded evidence and calculation results. It cites `EV-*`, `CALC-*`, and snapshot IDs, labels assumptions, and refuses fabrication, statutory-compliance requests, and automatic-control requests. A provider-backed adapter should preserve these same grounding and refusal rules.

## Deployment: free MVP path

The application is Vercel-compatible and has no required runtime services.

1. Push this repository to GitHub.
2. Create a free project at [vercel.com](https://vercel.com/) and import the repository.
3. Keep the framework preset as Next.js and the build command as `npm run build`.
4. Add `AI_PROVIDER=demo` if you want the setting to be explicit. Do not add an API key for the demo.
5. Deploy and open the generated URL at `/overview`.

For a local production check:

```bash
npm run build
npm run start
```

An optional container path is provided in [`Dockerfile`](./Dockerfile). It uses the standalone Next output and does not add a database or AI provider.

## Security, retention, and governance

CSV import is restricted to CSV files in the UI, limits the advertised file size to 10 MB, and treats all values as data rather than executable spreadsheet formulas/macros. The intended production boundary is server-side Zod validation, tenant-scoped repositories, secure cookies, rate limits around imports/AI, append-only audit events, and least-privilege roles. The demo is ephemeral and has no real personal or customer data; production deployments must define retention, deletion, backup, and tenant isolation policies before ingesting private evidence.

## Known limitations and next phases

The demo adapter does not persist browser edits after a reload, authenticate users, parse arbitrary CSV schemas on the server, connect to live industrial systems, or generate DOCX/XLSX. The seeded data is fictional and intentionally includes visible gaps and variance. Next phases are PostgreSQL/Drizzle persistence, authenticated tenant isolation, server-side import validation/storage, Playwright coverage, provider-backed grounded AI, approved snapshot persistence, and field-validation workflows for live read-only integrations.

`npm audit --audit-level=high` currently reports two moderate transitive PostCSS findings through the Next.js dependency tree and no high/critical findings. `npm audit fix --force` proposes an unsafe major downgrade, so it has not been applied; review the upstream Next/PostCSS advisory before production use.
